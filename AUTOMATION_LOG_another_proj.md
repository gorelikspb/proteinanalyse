# Лог автоматизации процессов разработки

Этот файл документирует решения по автоматизации рабочих процессов, которые применяются ко всем проектам.

---

## 🗓️ История автоматизации

### Этап 1: Автоматизация создания скриншотов

**Что было**: Чтобы вести дневник проекта, нужны регулярные скриншоты страниц. Делать их вручную долго и не гарантирует единый стиль.

**Решение**: Добавлен простой Node-скрипт на Puppeteer (`scripts/capture-screenshots.js`), который:
1. Открывает локальный `index.html`
2. Делает скриншоты каждого из нужных блоков
3. Сохраняет их в `screenshots/01-*.png ... 05-*`
4. Снимает отдельный полный скриншот мобильной версии

**Почему такое решение**:
- Puppeteer легко запускается командой `npm run capture:screens`
- Работает с локальным файлом — не нужен сервер
- Быстро повторно обновить скриншоты после изменений
- Сразу выдаёт все нужные варианты (desktop + mobile)

**Плюсы**:
- ✅ Экономит время, минимизирует ручные действия
- ✅ Скриншоты всегда в едином разрешении и стиле
- ✅ Можно запускать сколько угодно раз
- ✅ Не требует дополнительных внешних сервисов

**Минусы**:
- ❌ Нужна установленная Node.js
- ❌ Puppeteer весит ~100 МБ (Chromium)
- ❌ Возможны отличия при обновлениях браузера

**Подводные камни**:
- Для Windows важно запускать из корня проекта (используется `file://`)
- При первом запуске Puppeteer скачивает Chromium — нужно интернет-соединение
- Если меняются селекторы секций, нужно обновлять скрипт

**Что сделано**:
- ✅ Инициализирован `npm` и добавлен `puppeteer`
- ✅ Создан скрипт `scripts/capture-screenshots.js`
- ✅ Добавлена команда `npm run capture:screens`
- ✅ Скрипт сохраняет 5 скриншотов в соответствии с требованиями дневника

### Этап 2: Подготовка репозитория к публикации

**Что было**: Понадобилось выложить проект на GitHub, но в репозитории находились внутренние инструкции, журналы и скриншоты, которые не должны быть публичными.

**Решение**:
1. Инициализировали git-репозиторий и сделали первый коммит.
2. Добавили `.gitignore`, исключив `instructions/`, `screenshots/`, `PROJECT_LOG.md`, `node_modules/`, утилиты (`scripts/`) и прочие служебные файлы.
3. Удалили из истории все приватные материалы и запушили только продуктовый код.
4. Зафиксировали в `instructions/GITHUB_SETUP.md`, что служебные файлы всегда остаются локально.

**Почему такое решение**:
- Хотим оставлять на GitHub только то, что необходимо пользователям (лендинг и связанный код).
- Инструкции и журналы содержат рабочие заметки и должны храниться локально.
- Удаление лишнего облегчает размер репо и защищает внутренние процессы.

**Плюсы**:
- ✅ Репозиторий чистый, подходит для публичного просмотра.
- ✅ Нет риска случайно раскрыть служебную информацию.
- ✅ Простая инструкция для последующих проектов.

**Минусы**:
- ❌ Нужно синхронизировать инструкции вручную между проектами.
- ❌ Скриншоты и логи не доступны онлайн (только локально).

**Подводные камни**:
- Следить, чтобы `.gitignore` всегда обновлялся при появлении новых служебных файлов.
- При коммитах удостоверяться, что `git status` пуст от приватных данных.

**Что сделано**:
- ✅ Добавлен `.gitignore` с нужными исключениями.
- ✅ Репозиторий проинициализирован и запушен на GitHub.
- ✅ В инструкциях описано, что публикуем, а что оставляем локально.

### Этап 3: Централизованный радар проектов

**Что было**: Нужно агрегировать прогресс по всем проектам в одном месте (Notion/Airtable), чтобы видеть статусы и следующие шаги.

**Решение**:
1. Перешли с Markdown-таблицы на `PROJECTS_OVERVIEW.json` — структурированный формат, который легко читать скриптами.
2. Добавили файл `instructions/CENTRAL_RADAR_SETUP.md` с инструкцией, как собрать данные в отдельном репозитории.
3. Подготовили пример скрипта `collect.js`, который скачивает `PROJECTS_OVERVIEW.json` из всех репозиториев и сохраняет агрегированный `dashboard.json`.

**Почему такое решение**:
- JSON проще парсить и автоматически синхронизировать.
- Статичный Node.js скрипт можно запустить локально, а позже перенести в GitHub Actions или Airtable Automation.
- Центральный радар помогает планировать работу по всем проектам.

**Плюсы**:
- ✅ Унифицированный формат для всех репозиториев.
- ✅ Быстрый способ собрать сводку без ручного копирования таблиц.
- ✅ Есть инструкция, как разворачивать отдельный агрегатор.

**Минусы**:
- ❌ Пока нужно вручную запускать скрипт (не подключено к API Notion/Airtable).
- ❌ Требуется поддерживать список репозиториев (`repos.json`) актуальным.

**Подводные камни**:
- Проверять, что `PROJECTS_OVERVIEW.json` всегда коммитится вместе с изменениями.
- Если репозиторий приватный и файл недоступен по raw-ссылке, нужно подумать о токенах или исключениях.

**Что сделано**:
- ✅ Создан `PROJECTS_OVERVIEW.json` с актуальным статусом Printacopy.
- ✅ Обновлены инструкции (`PROJECTS_OVERVIEW_GUIDE.md`, `AI_INSTRUCTIONS.md`, `CENTRAL_RADAR_SETUP.md`, `GITHUB_SETUP.md`).
- ✅ Запланирован отдельный репозиторий для центрального радара.

## 🎯 Ключевые решения

### 1. Разделение публичного и приватного контента

**Решение**: Чёткое разделение на то, что идёт в GitHub (только продуктовый код), и то, что остаётся локально (инструкции, логи, скриншоты, утилиты).

**Обоснование**:
- ✅ Репозитории остаются чистыми и понятными для внешних пользователей
- ✅ Внутренние процессы не раскрываются
- ✅ Легко копировать структуру в новые проекты

```

├── instructions/          # Инструкции для Cursor/AI
```

```
project/
├── index.html             # Продуктовый код

├── script.js
```
