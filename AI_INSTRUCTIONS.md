# Инструкции для AI-ассистента

## 📋 О дневнике проекта

В этом проекте ведется **дневник проекта** (`PROJECT_LOG.md`), который документирует:
- Историю разработки и принятые решения
- Обоснование архитектурных выборов
- Анализ решений (плюсы, минусы, подводные камни)
- Уроки и выводы

## 🎯 Твоя роль при работе с дневником

### Когда добавлять записи в дневник:

1. **При принятии важных решений**
   - Выбор технологии/фреймворка/библиотеки
   - Архитектурные решения
   - Изменение подхода к решению задачи

2. **При выборе между альтернативами**
   - Когда рассматриваешь несколько вариантов
   - Объясни, почему выбрал один, а не другой

3. **При решении проблем**
   - С какими проблемами столкнулись
   - Как решили
   - Почему именно так

### Формат записи в дневник:

```markdown
### Этап X: [Название этапа]

**Что было**: [Описание исходной ситуации/проблемы]

**Решение**: [Что было сделано]

**Почему такое решение**:
- [Обоснование 1]
- [Обоснование 2]

**Плюсы**:
- ✅ [Плюс 1]
- ✅ [Плюс 2]

**Минусы**:
- ❌ [Минус 1]
- ❌ [Минус 2]

**Подводные камни**: 
- [Что учитывали]
- [Чего избежали]
```

## 📝 Что записывать

### ✅ Обязательно записывать:

- **Архитектурные решения** - выбор технологий, структура проекта
- **Компромиссы** - когда выбирали между вариантами
- **Проблемы и решения** - сложные баги, их решения
- **Изменения подхода** - когда меняли направление разработки

### ❌ Не обязательно:

- Мелкие багфиксы
- Очевидные решения
- Рутинные задачи

## 🔄 Как обновлять дневник

1. **Добавляй записи сразу** - пока помнишь детали решения
2. **Будь честным** - записывай не только успехи, но и проблемы
3. **Анализируй** - не просто описывай, а объясняй почему
4. **Обновляй разделы**:
   - "Будущие улучшения" - при появлении новых идей
   - "Метрики" - при изменении производительности
   - "Уроки и выводы" - в конце этапов

## 💬 Как общаться с пользователем о дневнике

Если пользователь спрашивает о решениях или хочет понять, почему что-то сделано так:
- **Ссылайся на дневник** - "Как записано в PROJECT_LOG.md, это решение было принято потому что..."
- **Предлагай обновить** - "Можем добавить это решение в дневник проекта"
- **Используй для объяснений** - дневник помогает объяснить контекст решений

## 🎨 Примеры записей

### Пример 1: Выбор технологии

```markdown
### Этап 3: Выбор фронтенд-фреймворка

**Что было**: Нужно было выбрать между React, Vue и vanilla JS

**Решение**: Выбрали vanilla JavaScript без фреймворков

**Почему такое решение**:
- Проект простой, не нужна сложная реактивность
- Быстрая загрузка (<200ms)
- Нет зависимостей

**Плюсы**:
- ✅ Быстрая загрузка
- ✅ Простота деплоя
- ✅ Нет build процесса

**Минусы**:
- ❌ Больше кода для написания
- ❌ Нет готовых компонентов

**Подводные камни**: 
- Нужно было убедиться, что функционал не перерастет в сложность
- Проверили, что vanilla JS справится со всеми задачами
```

### Пример 2: Решение проблемы

```markdown
### Этап 5: Решение проблемы с rate limiting

**Что было**: Google Trends API возвращал ошибку 429 (rate limit)

**Решение**: Добавили кэширование результатов и увеличили задержки между запросами

**Почему такое решение**:
- Кэширование снижает количество запросов
- Задержки помогают избежать блокировки
- Не требует изменения архитектуры

**Плюсы**:
- ✅ Решает проблему rate limiting
- ✅ Улучшает UX (быстрые повторные запросы)
- ✅ Не усложняет код

**Минусы**:
- ❌ Результаты могут быть устаревшими (TTL 1 час)
- ❌ Нужно управлять размером кэша

**Подводные камни**: 
- Важно было не переусердствовать с задержками (слишком медленно)
- Нужно было балансировать между свежестью данных и количеством запросов
```

## 🚨 Важные правила

1. **Всегда объясняй "почему"** - не просто "сделали X", а "сделали X потому что Y"
2. **Записывай альтернативы** - что еще рассматривали и почему не выбрали
3. **Будь честным о минусах** - не скрывай недостатки решений
4. **Фиксируй подводные камни** - что учитывали, чего избежали
5. **Обновляй регулярно** - не откладывай на потом
