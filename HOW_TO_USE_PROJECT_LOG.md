# Как использовать дневник проекта

## 📖 Что это такое?

Дневник проекта (`PROJECT_LOG.md`) - это документ, который фиксирует:
- **Историю разработки** - этапы, решения, изменения
- **Обоснование решений** - почему было выбрано то или иное решение
- **Анализ решений** - плюсы, минусы, подводные камни
- **Уроки** - что сработало, что можно улучшить

## 🚀 Быстрый старт

### Шаг 1: Скопируйте шаблон

В начале нового проекта скопируйте `PROJECT_LOG_TEMPLATE.md` в `PROJECT_LOG.md`:

```bash
cp PROJECT_LOG_TEMPLATE.md PROJECT_LOG.md
```

Или просто создайте новый файл `PROJECT_LOG.md` и скопируйте содержимое шаблона.

### Шаг 2: Заполните обзор

Замените в `PROJECT_LOG.md`:
- `[Название проекта]` → реальное название
- `[Краткое описание цели проекта]` → описание цели
- `[Список используемых технологий]` → используемые технологии
- `[Где и как деплоится]` → информация о деплое

### Шаг 3: Ведите дневник по ходу разработки

**Когда добавлять записи:**
- При принятии важных архитектурных решений
- При изменении подхода к решению задачи
- При выборе между альтернативами
- При обнаружении проблем и их решении

**Что записывать:**
1. **Что было** - исходная ситуация/проблема
2. **Решение** - что было сделано
3. **Почему** - обоснование решения
4. **Плюсы/минусы** - анализ решения
5. **Подводные камни** - что учитывали, чего избежали

## 📝 Структура записей

### Пример записи этапа:

```markdown
### Этап 1: Выбор технологии для бэкенда

**Что было**: Нужно было выбрать технологию для API

**Решение**: Выбрали FastAPI вместо Flask

**Почему такое решение**:
- FastAPI быстрее и современнее
- Автоматическая документация
- Type hints из коробки

**Плюсы**:
- ✅ Высокая производительность
- ✅ Современный подход
- ✅ Хорошая документация

**Минусы**:
- ❌ Меньше примеров в интернете
- ❌ Более новая технология

**Подводные камни**: 
- Нужно было убедиться в стабильности
- Проверили совместимость с существующим кодом
```

## 🎯 Когда вести дневник

### ✅ Обязательно записывать:

1. **Архитектурные решения**
   - Выбор фреймворка/библиотеки
   - Структура проекта
   - Паттерны проектирования

2. **Компромиссы**
   - Когда выбирали между вариантами
   - Почему выбрали один, а не другой

3. **Проблемы и решения**
   - С какими проблемами столкнулись
   - Как решили
   - Почему именно так

4. **Изменения подхода**
   - Когда меняли направление
   - Почему пришлось изменить

### ❌ Не обязательно записывать:

- Мелкие багфиксы
- Очевидные решения
- Рутинные задачи

## 📸 Скриншоты

### Как добавить скриншоты:

1. Создайте папку `screenshots/` в корне проекта
2. Сохраняйте скриншоты с понятными именами:
   - `01-main-page.png`
   - `02-feature-x.png`
   - и т.д.
3. Добавьте ссылки в дневник:
   ```markdown
   ![Описание](screenshots/01-main-page.png)
   ```

### Важно:
- Если проект публичный, скриншоты могут быть в репозитории
- Если скриншоты не должны публиковаться, добавьте `screenshots/` в `.gitignore`
- Или храните скриншоты отдельно от публичного кода

## 🔄 Обновление дневника

### Регулярность:
- **Во время разработки**: добавляйте записи по мере принятия решений
- **В конце спринта/этапа**: подведите итоги
- **Перед релизом**: добавьте финальные метрики и выводы

### Что обновлять:
- Добавлять новые этапы
- Обновлять метрики
- Добавлять уроки и выводы
- Обновлять раздел "Будущие улучшения"

## 💡 Советы

1. **Пишите сразу** - не откладывайте, пока помните детали
2. **Будьте честны** - записывайте не только успехи, но и проблемы
3. **Анализируйте** - не просто описывайте, а анализируйте решения
4. **Используйте для обучения** - дневник помогает понять, что сработало, а что нет

## 📚 Примеры использования

### Для личных проектов:
- Фиксация решений для будущего себя
- Обучение на собственном опыте
- Портфолио с объяснением решений

### Для командных проектов:
- Обмен знаниями в команде
- Онбординг новых разработчиков
- Документация принятых решений

### Для open-source:
- Прозрачность процесса разработки
- Объяснение архитектурных решений
- История проекта

## 🎨 Кастомизация

Шаблон можно адаптировать под свои нужды:
- Добавить разделы специфичные для вашего типа проектов
- Изменить структуру под свой стиль
- Добавить метрики, важные для вашего проекта

---

**Главное**: Дневник должен быть полезным для вас и других, кто будет работать с проектом. Пишите так, чтобы через месяц/год было понятно, почему были приняты те или иные решения.
